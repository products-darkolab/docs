## How it works

Safua uses a custom migration runner on top of Alembic. Every new feature adds a migration file to `api/migrations/` and registers it in `MIGRATION_ORDER` in `run_migrations.py`.

Migrations run automatically on Railway deploy. Locally, run them manually:

```bash
cd api
python run_migrations.py
```

## Writing a Migration

Create a new file in `api/migrations/`:

```python
# api/migrations/add_my_feature.py

from sqlalchemy.ext.asyncio import create_async_engine
from sqlalchemy import text
from app.config import get_settings

DATABASE_URL = get_settings().DATABASE_URL

STATEMENTS = [
    "ALTER TABLE users ADD COLUMN IF NOT EXISTS my_column VARCHAR(50)",
    """
    CREATE TABLE IF NOT EXISTS my_table (
        id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
        user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
        created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
    )
    """,
    "CREATE INDEX IF NOT EXISTS idx_my_table_user ON my_table(user_id)",
]

async def run():
    engine = create_async_engine(DATABASE_URL, echo=False)
    async with engine.begin() as conn:
        for stmt in STATEMENTS:
            stmt = stmt.strip()
            if stmt:
                await conn.execute(text(stmt))
    await engine.dispose()
    print("  Added my feature tables")

if __name__ == "__main__":
    import asyncio
    asyncio.run(run())
```

Then register it in `run_migrations.py`:

```python
MIGRATION_ORDER = [
    ...
    "add_cohort_matching",
    "add_my_feature",    # ← add here, after all existing migrations
]
```

## Rules

<Warning>
**Never use multiple SQL statements in a single `text()` call.** asyncpg doesn't support it. Always split on `;` and execute each statement separately. The `STATEMENTS` list pattern above handles this correctly.
</Warning>

1. **Always use `IF NOT EXISTS`** — migrations must be idempotent. Re-running should never fail.
2. **Never drop columns or tables** in a migration unless you're 100% sure no code references them.
3. **Register in order** — the order in `MIGRATION_ORDER` is the execution order. New migrations go at the bottom.
4. **Test locally before pushing** — run `python run_migrations.py` locally with a fresh DB to verify.
